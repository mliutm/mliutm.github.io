什么是SSM框架，SSM框架的作用

1.框架的概念：框架：在这里特指软件框架，它是我们在实际开发中解决项目需求的技术集合。运用框架可以大大简化我们的代码编写，缩短开发周期。同时，对后续项目维护在面对开发团队人员更迭时降低了门槛，对系统升级提供灵活可控的标准。当然，框架提高了我们的开发效率，降低维护成本的同时，也要求我们在使用时遵循它的规范，要按照它的使用要求编写代码。最后，需要注意的是，它不是系统架构。

2.框架的作用：提高开发效率 ，增强可重用性， 节约维护成本，提供编写规范 ，解耦底层实现原理。

3.SSM框架：SSM主要构成Spring，SpringMVC，Mybatis三大部分组成

spring、springmvc、mybatis各个组件的核心功能

spring核心功能

  • IOC和AOP是spring的核心

1.IOC

1.1IOC的定义

IOC,控制反转，它是一个技术思想，核心就是将new对象的操作交由IOC容器去帮助我们完成，包括创建实例化对象并管理它，我们需要使用哪个对象，去IOC容器获取即可。控制指的是对象创建（实例化、管理）的权力，反转指的是控制权交给了外部环境（IOC容器）

1.2IOC解决了什么问题

对象之间的耦合问题

1.3IOC和DI的区别

IOC和DI描述的是同一件事情（对象实例化以及依赖关系维护这件事），只是不同角度罢了。

IOC是站在对象的角度，对象实例化及其管理的权力交给了容器，DI是站在容器的角度，容器会把对象依赖的属性注入。

2.AOP

2.1AOP定义

面向切面编程，是OOP（面向对象）的延续，OOP解决不了以下问题：

  • 代码重复问题
  • 逻辑代码和业务代码混杂在一起，代码臃肿，维护不方便

2.2.2逻辑代码和业务代码的区别

  • 逻辑代码是指实现程序功能的核心算法、逻辑或数据处理部分的代码，它通常与具体业务无关，是实现业务逻辑的基础。而业务代码则是指针对具体业务需求编写的代码，主要包括用户交互、数据输入输出、状态管理等方面的实现。简单来说，逻辑代码是通用的、可重用的，并负责解决问题的核心，而业务代码则是具体的、非通用的，负责实现业务需求。

AOP提出横向抽取机制，将横切逻辑代码和业务逻辑代码分离

2.2、AOP解决了什么问题

在不改变原有业务逻辑情况下，增强横切逻辑代码，根本上解耦合，避免横切逻辑代码重复。

2.3、为什么叫做面向切面编程

切：指的是横切逻辑，原有业务逻辑代码我们不能动，只能操作横切逻辑代码，所以⾯向横切逻辑

⾯：横切逻辑代码往往要影响的是很多个⽅法，每⼀个⽅法都如同⼀个点，多个点构成⾯，有⼀个⾯的概念在⾥⾯

  • Spring 为**表示层**提供的基于 MVC 设计理念的优秀的 Web 框架，是目前最主流的 MVC 框架之一。
  • 一种轻量级的、基于MVC的Web层应用框架。偏前端而不是基于业务逻辑层。

什么是jdbc

  • JDBC（Java Database Connectivity）是Java连接数据库的标准API，它提供了一种Java程序访问关系型数据库的标准方法。通过JDBC API，开发者可以使用Java程序与不同的关系型数据库（如MySQL、Oracle、DB2等）交互，执行SQL语句、获取查询结果等。JDBC API可以通过JDBC驱动来实现与具体的数据库之间的连接和交互。

spring、springmvc、mybatis各个组件的关系

三层架构

使用框架最重要的目的：解耦合

三层架构是指：视图层（表现层）View、业务逻辑层Service、数据访问层（持久层）Dao，它们分别完成不同的功能。

  • View层：用于接收用户提交请求的代码在这里编写。
  • Service层：系统的业务逻辑主要在这里完成。
  • Dao层：直接操作数据库的代码在这里编写。

==为了更好的降低各层间的耦合度，在三层架构程序设计中，采用面向抽象编程。即上层对下层的调用，是通过接口实现的。而下层对上层的真正服务提供者，是下层接口的实现类。服务标准（接口）是相同的，服务体供着（实现类）可以更换。这就实现了层间解耦合。==

什么是MVC？

MVC，即Model模型、View视图、及Controller控制器

  • **View:**视图，为用户提供使用界面，与用户直接进行交互。
  • Model:模型，承载数据，并对用户提交请求进行计算的模块。其分为两类：一类称为数据承载bean，一类称为业务处理bean。所谓数据承载bean是指实体类，专门为用户承载业务数据的，如 Student、User等。而业务处理bean则是指Service或Dao对象，专门用来处理用户提交请求的。
  • Controller:控制器，用于将用户请求转发给相应的Model进行处理，并根据Model的计算结果向用户提供相应的响应。

MVC架构程序的工作流程是这样的：

  1. 用户通过View页面向服务端提出请求，可以是表单请求、超链接请求、AJAX 请求等。
  2. 服务端Controller控制器接收到请求后对请求进行解析，找到相应的Model对用户进行处理。
  3. Model处理后，将处理结果再交给Controller。
  4. Controller在接收到处理结果后，根据处理结果找到要作为向客户端发回的响应View页面。页面经过渲染（数据填充）后，再发送给客户端。

SSM与三层架构的关系

SSM，即SpringMVC、Spring与MyBatis三个框架。它们再三层架构中所处的位置是不同的。即它们再三层架构中的功能各不相同，各司其职。

SpringMVC:作为View层的实现者，完成用户的请求与接收功能。SpringMVC的Controller作为整个应用的控制器，完成用户请求的转发及对用户的响应。

**MyBatis:**作为Dao层的实现者，完成对数据库的增、删、改、查功能。

Spring：以整个应用大管家的身份出现。整个应用中所有Bean的声明周期行为，均由Spring来管理，即整个应用中所有对象的创建、初始化、销毁、及对象间关联关系的维护，均由Spring进行管理。

spring和springmvc需要整合吗？

  1. Spring 和 SpringMVC 都是用来管理对象的容器。
  2. **SpringMVC 主要用来管理 Controller ，Spring 主要用来管理 Service 和 Dao 以及其他对象。**所以我们在 SpringMVC 的配置文件里配置的注解扫描路径就是 Controller的路径，而 Spring 的配置文件配置的注解扫描路径会排除 Controller。
  3. Spring 和 SpringMVC 是父子容器的关系。Spring 容器是父容器，SpringMVC 是子容器。
  4. 子容器可以访问父容器的对象，但是在父容器里不能访问子容器的对象。也就是说在 Controller 里可以访问 Service 对象，但是在 service 里不可以访问 Controller 对象。但是，需要注意的是，可以通过在Spring配置文件中将Spring MVC容器中的bean定义为父容器中的bean来解决这个问题。这样做可以使得Spring容器中的bean也能够访问Spring MVC容器中的bean。具体实现方式可以参考Spring文档中有关父子容器的说明。
  5. SpringMVC 是 Spring 的子项目，属于 Spring 家族的一部分。所以在潜意识里，你可以认为他俩几乎是黏在一起的。

spring配置文件中需要配置的组件，以及配置的各组件的功能

  1. 在applicationContext.xml文件中需要配置以下组件：

    1. 数据库连接池：配置数据源，包括数据库驱动、数据库URL、用户名、密码等信息，以便在应用程序启动时创建并管理数据库连接池。
    2. MyBatis的SqlSessionFactoryBean：将MyBatis框架与Spring框架整合，提供一个SqlSessionFactory实例，用于创建SqlSession对象。
    3. MapperScannerConfigurer：自动扫描指定包下的Mapper接口，并将其注册为Spring的Bean，使得可以在其他地方使用@Autowired注解来注入Mapper对象。
    4. TransactionManager：配置事务管理器，将事务管理交给Spring框架统一处理。
    5. 扫描注解：使用context:component-scan标签扫描指定包下的组件，并将其注册为Spring的Bean，简化配置过程。

    这些组件的功能分别如下：

    1. 数据库连接池：负责管理数据库连接，避免频繁地创建和销毁数据库连接，提高应用程序的性能和稳定性。
    2. MyBatis的SqlSessionFactoryBean：将MyBatis框架与Spring框架整合，提供一个SqlSessionFactory实例，用于创建SqlSession对象，同时也支持对MyBatis的配置进行定制。
    3. MapperScannerConfigurer：自动扫描指定包下的Mapper接口，并将其注册为Spring的Bean

springmvc配置文件中需要配置的组件，以及配置的各组件的功能

在Spring MVC的配置文件中，需要配置以下组件：

  1. HandlerMapping：将URL映射到处理程序方法。
  2. ViewResolver：将逻辑视图名称解析为实际视图对象。
  3. MessageSource：用于国际化（i18n）和本地化（l10n），提供资源束和信息查找服务。
  4. DataBinder：将HTTP请求参数绑定到JavaBean中的属性。
  5. Validator：用于验证表单数据。
  6. Interceptor：拦截处理程序执行链并允许在处理程序处理请求之前或之后应用自定义逻辑。
  7. HandlerAdapter：将处理程序包装成适当的接口以与DispatcherServlet交互。

这些组件的功能如下：

  1. HandlerMapping：确定哪个控制器处理特定的请求。
  2. ViewResolver：将逻辑视图名称解析为实际的视图对象，例如JSP或Thymeleaf模板。
  3. MessageSource：提供国际化和本地化支持，可以根据请求头信息返回对应语言版本的文本。
  4. DataBinder：将HTTP请求参数绑定到JavaBean中的属性，从而构造出controller需要的ModelAndView实例。
  5. Validator：验证表单数据是否符合预期，如果验证失败则能够提示用户错误信息。
  6. Interceptor：通过定义preHandle()、postHandle()和afterCompletion()方法，在请求的不同阶段进行一些处理操作，例如身份验证、日志记录等。
  7. HandlerAdapter：将处理程序包装成适当的接口以与DispatcherServlet交互，使得controller能够正确处理各种类型的请求。

mybatis配置文件中需要配置的组件，以及配置的各个组件的功能

在整合SSM框架中，mybatis配置文件需要配置以下信息：

  3. 映射器（mapper）：指定MyBatis映射器接口的路径和对应的SQL映射文件的位置。

其中，SQL映射文件中需要配置SQL语句及参数映射规则。

SSM整合主要是整合什么？

首先我们都知道，SpringMvc是Spring的一个后续产品，SpringMvc和Spring是子容器与父容器的关系，其实它们整合与否都可，整合它们也只是避免业务逻辑混乱。让SpringMvc处理视图层，Spring处理来处理具体的业务逻辑。所以SSM真正需要整合的，是Spring和Mybatis之间的整合。

“MVC:view-controller” 标签通常是用于 web 应用程序中的模型-视图-控制器 (MVC) 框架中。它可以帮助将视图和控制器分离，从而使代码更加模块化和易于维护。该标签指定了一个视图的名称以及与该视图关联的控制器。当浏览器请求该视图时，框架可以自动查找并调用相应的控制器来处理该请求并显示正确的视图。
