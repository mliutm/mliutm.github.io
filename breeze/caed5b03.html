3 服务改造

3.1 分支策略

3.1.1 弱得批爆仓库策略-只有一个主分支

​ 只有一个主分支,比较弱得是,谁都随时可提! 稍微好一点,只有项目经理可以提,你开发完后找项目经理检视代码后,项目经理用他的账号提交.

​ 优点: 简单

​ 缺点: 很容易冲突,不好做产品定制开发

3.1.2 独立项目分支策略

​ 我们的仓库使用gitee,我们有一个主分支(测试了没有问题,通过打tag发布版本),每个开发人员都会在自己特性分支上面开发代码,开发完毕后,找项目经理进行代码检查,如果ok后.在gitee上面提交pull request请求,项目经理合并代码!

3.1.3 产品模型分支

产品公司:

​ 卖crm—>产品crm(master)—-蓉华crm(crm-mliutm),dbl公司crm(crm-dbl)—->feature

​ 基础技术平台(任何系统都有功能)(master)—开发其他业务(crm,car)—–蓉华crm,dbl公司crm(crm-mliutm)—->feature

============================

​ mliutm-plaform—>crm/car—feature: 项目思维

​ mliutm-plaform—>crm/car—->mliutm-crm/dbl-crm—feature:产品思维

================================================

3 项目实战

3.1 阿里云oss上传方式-web

​ 1)前端直接传阿里云

​ 这种方式意味着需要把OSS配置参数写死在JS中，有安全隐患。

String endpoint = “https://oss-cn-hangzhou.aliyuncs.com/“;
// 阿里云主账号AccessKey拥有所有API的访问权限，风险很高。强烈建议您创建并使用RAM账号进行API访问或日常运维，请登录RAM控制台创建RAM账号。
String accessKeyId = “LTAI5tAX4GY4ZTHTtcMX2D43”;
String accessKeySecret = “QMwsjZRR6dqa9Y4mW24S4JVhHcZOAD”;
String bucketName = “yaosang-gift-20220516”;

​ 好处：降低后端服务器压力 坏处：不安全

​ 2)前端调用我们后端接口，后端上传阿里云—上个项目

​ AliyunOssTest这种代码已经把参数封装后端。

​ 好处：安全 坏处:增加了后端服务器压力,接收图片在上传oss。

​ 3） 后端签名+前端上传（携带签名）—采纳

​ 签名接口其实只需要接收几个字符串参数，响应字符串值就ok了，不像原来后端传递要接收很大的图片或者视频等流。可以认为此时的后端没有任何损耗。

​ 后端需要写获取签名的接口？？？？写在哪儿？ 可以写一个阿里云oss微服务服务，同样短信服务 ，邮件服务，图片验证码服务，也是一个单独的服务。造成一个问题？ 服务太多。

​ 写一个公共服务，就包含了这些小的功能。
